# Backend Scaffolding Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Restructure the repo into an npm-workspaces monorepo, move the existing Vue client into `packages/client/`, add a Fastify+ws server scaffold in `packages/server/`, add a types-only `packages/shared/`, and add Docker + Vercel config — all without breaking the existing client or the existing Vercel deploy.

**Architecture:** Single feature branch `feature/backend-scaffolding` off `main` (currently `9b1c8c6`). Each task in this plan gets its own sub-branch (named `task/NN-<slug>`) that merges into `feature/backend-scaffolding` with `--no-ff` once its verification gates are green. The scaffold contains zero sync protocol logic — that work happens in a separate spec/plan cycle.

**Tech Stack:** Vue 3 + TS + Vite + Vitest (client, unchanged behavior), Fastify 5 + @fastify/websocket 11 + tsx + Vitest (server), TypeScript-only types-shared package, multi-stage Dockerfile + docker-compose for local dev, npm workspaces, Vercel (client) + Render (server, deploy is follow-up).

## Source-of-truth context the implementer needs

Before any task, know these things about the current repo (`main` at `9b1c8c6`):

- Single package at the repo root. Files at root: `index.html`, `package.json`, `package-lock.json`, `tsconfig.json`, `tsconfig.node.json`, `vite.config.ts`, `src/`, `docs/`, `dist/`, `node_modules/`, `screenshot.png`.
- `src/` contains: `App.vue`, `main.ts`, `env.d.ts`, plus subdirectories `components/`, `composables/`, `engine/`, `project/`, `sequencer/`, `utils/`.
- All cross-module imports use relative paths (`./project`, `../engine/...`). `src/project/index.ts` is the public barrel that re-exports `Project`, `ProjectTrack`, `EngineType`, `MixerState`, `DEFAULT_MIXER_STATE`, `PROJECT_SCHEMA_VERSION`, `activeParams`, etc. Almost every consumer of project types goes through this barrel.
- `src/project/types.ts` (the file the spec mentions as a "byte-for-byte copy" candidate for shared) imports engine param types from `../engine/SynthEngine`, `../engine/KickEngine`, etc. **This means a literal byte-for-byte copy of `types.ts` into `packages/shared/` would break** — `../engine/...` does not resolve from the shared package. The plan handles this by exporting only the truly portable symbols from `@fiddle/shared` for this PR (see Task 3 notes). The fuller type unification is deferred to the sync brainstorm.
- Baseline: 182 tests passing, `vue-tsc --noEmit` clean, `vite build` clean.
- No `vercel.json` exists; Vercel currently auto-detects Vite from the root `package.json`.

## Branch and merge conventions

- Sub-branches off `feature/backend-scaffolding`, named `task/NN-<slug>` (e.g. `task/02-restructure-client`).
- After all verification commands in a task pass, merge with `git merge --no-ff task/NN-<slug>` into `feature/backend-scaffolding`. **Do not delete the sub-branch** until the whole feature merges to `main` (the user wants merge history visible for review).

export interface EngineParamsMap {
  synth: SynthEngineParams;
  kick: KickEngineParams;
  hat: HatEngineParams;
  snare: SnareEngineParams;
  clap: ClapEngineParams;
}

export interface ProjectTrack {
  engineType: EngineType;
  engines: EngineParamsMap;
  mixer: MixerState;
  steps: Step[];

export interface Project {
  schemaVersion: 1;
  bpm: number;
  tracks: [ProjectTrack, ProjectTrack, ProjectTrack, ProjectTrack];
}

export function activeParams<T extends EngineType>(

  return track.engines[track.engineType] as EngineParamsMap[T];

export async function wsRoute(app: FastifyInstance) {

## Known divergences from the spec

- **`packages/shared/src/project-types.ts` is not created.** The spec describes it as a "byte-for-byte copy of `src/project/types.ts`", but that copy doesn't compile because `types.ts` imports engine param types from `../engine/SynthEngine` etc., which don't resolve from outside the client. Instead, Task 3 puts only the 4 truly portable symbols (`EngineType`, `MixerState`, `DEFAULT_MIXER_STATE`, `PROJECT_SCHEMA_VERSION`) directly into `packages/shared/src/index.ts`, and the client re-exports them. The fuller type unification (engine param types, `Project`, `ProjectTrack`) is deferred to the sync brainstorm where it can be designed properly (likely by extracting type-only declarations from engine modules into shared). The spec's goal — `@fiddle/shared` exists and the client imports from it — is satisfied either way.

This divergence is intentional. The spec's stated goal ("@fiddle/shared exists; client imports from it; future sync work plugs in without restructure") is satisfied. If the divergence needs to be reflected back into the spec doc, do it as a follow-up edit on the spec — do not let it expand the scope of this scaffold PR.
